## 1. Документация и спецификации

### 1.1 Несоответствие количества удалённых строк

**Файл:** `dataset_card.md`  
**Проблема:** В разделе "Data quality summary" указано `Rows dropped merging → processed: **28**`, однако фактическое количество: 2435 (merged) − 2406 (processed) = **29**.  
**Как исправить:** Заменить `28` на `29` в `dataset_card.md`.

### 1.2 Расхождение estimated_records в source_map.json и реального числа

**Файл:** `specs/source_map.json`  
**Проблема:** Поля `estimated_records` для ряда источников существенно расходятся с реальными числами:

| source_id | estimated_records | Фактически в dataset.csv |
|-----------|-------------------|--------------------------|
| `paper_zhang_2024` | 40 | 10 |
| `paper_lee_2023` | 40 | 16 |
| `paper_melittin_processes_mdpi` | 80 | 34 |
| `paper_b7_proline_rich_2025` | 9 | 6 |

**Как исправить:** Обновить `estimated_records` на актуальные значения, либо переименовать поле в `actual_records` после завершения экстракции. Для завершённого проекта оценочные числа стоит заменить реальными.

### 1.3 Validation rules не полностью реализованы в коде

**Файлы:** `specs/validation_rules.json`, `scripts/validate_project.py`  
**Проблема:** Правило `measurement_value_not_null` (severity: error) описано в спецификации, однако в `validate_project.py` функция `check_measurement_value` **пропускает** null-значения вместо того, чтобы считать их ошибкой:
```python
if pd.isna(val) or val == "":
    continue  # ← должно быть issues.append(...)
```
Аналогично, правило `peptide_sequence_not_null` не реализовано как отдельная проверка (хотя по факту nullов нет).  
**Как исправить:** Добавить явную проверку на null для `measurement_value` и `peptide_sequence` в `validate_project.py`, чтобы соответствовать декларированным правилам.

## 2. Качество кода

### 2.1 Пять строк с null measurement_value проходят валидацию

**Файл:** `data/processed/dataset.csv`  
**Проблема:** В итоговом датасете 5 строк (все из `paper_sk_peptides_springer_2025`) содержат пустой `measurement_value`, хотя схема (`dataset_schema.json`) объявляет это поле обязательным (`"required": true`). Скрипт `clean_dataset.py` не удаляет строки с null MIC после нормализации.  
**Как исправить:** В `clean_dataset.py`, после вызова `normalize_verbatim_mic`, добавить фильтрацию:
```python
if "measurement_value" in out.columns:
    mv_empty = out["measurement_value"].isna() | (out["measurement_value"] == "")
    if mv_empty.any():
        out = out.loc[~mv_empty].copy()
```
Затем перегенерировать `dataset.csv`.

### 2.2 Функция `parse_numeric_measurement` не вызывается напрямую

**Файл:** `scripts/clean_dataset.py`  
**Проблема:** Функция `parse_numeric_measurement` определена отдельно, но используется только внутри `normalize_float`. Это не мёртвый код, но нарушает принцип минимального API — можно было инлайнить логику в `normalize_float`.  
**Как исправить:** Незначительное замечание; можно оставить как есть или объединить в одну функцию. Не критично.

### 2.3 Дублирование `load_schema_columns` в нескольких скриптах

**Файлы:** `scripts/build_dataset.py`, `scripts/clean_dataset.py`, `scripts/extract_pdf.py`, `scripts/extract_web.py`  
**Проблема:** Функция `load_schema_field_names()` / `load_schema_columns()` определена в 3-4 файлах с идентичной реализацией.  
**Как исправить:** Вынести в `scripts/utils.py` и импортировать оттуда. Незначительно, но улучшает поддерживаемость.

### 2.4 Нет проверки на `pathogen_name` not null в validate_project.py

**Файл:** `scripts/validate_project.py`  
**Проблема:** Поле `pathogen_name` объявлено required в схеме, но валидатор не проверяет его на null. По факту nullов нет, но проверка должна быть для полноты.  
**Как исправить:** Добавить `check_pathogen_name(df)` аналогично `check_record_id`.

## 6. Прочие замечания

### 6.1 Notebook `eda.ipynb` не содержит markdown-ячеек

**Файл:** `notebooks/eda.ipynb`  
**Проблема:** Все 9 ячеек — code cells. Нет поясняющих markdown-ячеек с заголовками, описаниями графиков или выводами. Для курсового проекта EDA-ноутбук должен быть самодостаточным и читаемым.  
**Как исправить:** Добавить markdown-ячейки с заголовком, описанием каждого визуала и кратким выводом в конце.

### 6.2 `requirements.txt` без версий пакетов

**Файл:** `requirements.txt`  
**Проблема:** Зависимости указаны без версий (`pandas`, `pytest`, `requests`, `openpyxl`, `pdfplumber`). Это снижает воспроизводимость — при изменении API библиотек скрипты могут перестать работать.  
**Как исправить:** Закрепить минимальные версии, например:
```
pandas>=2.0
pytest>=7.0
requests>=2.28
openpyxl>=3.1
pdfplumber>=0.10
```

### 6.3 `matplotlib` и `seaborn` не указаны в requirements.txt

**Файл:** `requirements.txt`  
**Проблема:** Ноутбук `eda.ipynb` импортирует `matplotlib` и `seaborn`, но они не указаны в `requirements.txt`.  
**Как исправить:** Добавить `matplotlib` и `seaborn` в `requirements.txt`.

### 6.4 Данные из extraction_log свидетельствуют о проблеме DBAASP

**Проблема:** Из DBAASP API извлечено 1000 записей (max_records=1000), но в итоговом датасете только 974. Разница (26 строк) потеряна при очистке/дедупликации. В документации (dataset_card, README) указано 974 как ожидаемое число, что корректно для финального результата, но `web_extraction_manifest.json` указывает `"max_records": 1000` — потенциально сбивает с толку.  
**Как исправить:** Добавить комментарий в source_map или dataset_card: "974 rows remain after cleaning; 1000 were extracted before filtering and deduplication."

### 6.5 Опечатки в commit messages

Несколько коммитов содержат опечатки (`traubles`, `abd`, `snall`, `deasription`) — непрофессионально для курсового проекта, но не влияет на содержание.
